# PostgreSQL Migration Plan (SQLite -> Postgres, Kysely, SQL Migrations)

## Summary
Migrate backend persistence from `better-sqlite3` to PostgreSQL with a full cutover, keeping existing route behavior and `database` module API stable for the rest of the app. Use Kysely + `pg`, versioned SQL migrations, and a manual one-time SQLite import script.

## Implementation Changes
1. **DB runtime refactor (no API contract change to routes)**
- Replace SQLite initialization in `server/src/db.ts` with a Postgres connection layer (`pg` pool + Kysely instance).
- Keep exported function names/signatures used by routes/tests unchanged (`getSettings`, `recordSession`, `createUser`, etc.) so route modules do not require behavioral rewrites.
- Convert all SQLite-specific SQL to Postgres-compatible SQL:
  - `INTEGER PRIMARY KEY AUTOINCREMENT` -> `GENERATED BY DEFAULT AS IDENTITY`
  - `INSERT OR IGNORE` -> `INSERT ... ON CONFLICT DO NOTHING`
  - `?` placeholders -> parameterized bindings via Kysely/sql templates
  - `DATE('now', '-13 days')` and SQLite date ops -> Postgres date/timestamp expressions
  - SQLite PRAGMA/migration helpers removed.

2. **Schema and migrations (versioned SQL)**
- Add migration system with versioned SQL files (e.g., `server/src/db/migrations/*.sql`) and a migration runner script.
- Baseline migration creates all current tables/constraints/indexes in Postgres equivalent form (`users`, `settings`, `progress`, `language_progress`, `category_progress`, `session_history`, `active_sessions`, `daily_xp`, `item_progress`, `attempt_history`, `email_verifications`, `password_resets`, `exercise_usage`, `community_exercises`).
- Add npm scripts for migration lifecycle (e.g., `db:migrate`, `db:migrate:status`).

3. **One-time SQLite import (manual command)**
- Add manual import script (`npm run db:import-sqlite`) that:
  - Reads SQLite file path (existing `LINGOFLOW_DB_PATH` or explicit import arg/env),
  - Extracts data table-by-table,
  - Inserts into Postgres in FK-safe order inside transactions/batches,
  - Is idempotent where possible (upsert/conflict handling),
  - Emits a summary report (rows read/inserted/skipped/errors).
- Import is not run automatically at server startup.

4. **Config, dependencies, docs, and tests**
- Replace server dependency on `better-sqlite3` with `pg` + `kysely`.
- Introduce Postgres env config (`DATABASE_URL`; optional SSL/pool envs) and update `server/.env.example`.
- Update README/Troubleshooting to reflect Postgres-first setup and import workflow.
- Update backend tests to require Postgres connection and isolate test data (per-test schema or truncation strategy), without adding repo-managed Docker/CI Postgres setup.

## Public Interfaces / Contract Impact
- **HTTP API:** no endpoint or payload shape changes expected.
- **Server env vars:**
  - Add: `DATABASE_URL` (required for server startup/tests).
  - Keep `LINGOFLOW_DB_PATH` only for one-time import input compatibility (not runtime DB).
- **NPM scripts:** add DB migration/import scripts for operational workflows.

## Test Plan
1. **Unit/integration parity**
- Run existing server tests against Postgres and ensure unchanged expected behavior for auth, session lifecycle, progression, and stats endpoints.

2. **Migration validation**
- Fresh DB: run migrations from empty database and verify all tables/constraints exist.
- Re-run migrations to confirm no-op/idempotent behavior.

3. **Import validation**
- Import from a representative SQLite DB snapshot.
- Verify row counts per table and key invariants (FK integrity, unique constraints, progress/streak totals).
- Smoke test critical flows after import: login, start/complete session, settings save, stats retrieval.

4. **Failure cases**
- Missing/invalid `DATABASE_URL` fails fast with actionable error.
- Import handles duplicates/conflicts safely and reports them.

## Assumptions and Defaults
- Full cutover to Postgres only (no SQLite runtime fallback, no dual-write).
- Kysely is the selected query builder.
- Versioned SQL migrations are the schema source of truth.
- One-time import is manual via npm script.
- No repo-provided local/CI Postgres orchestration; environment supplies Postgres.
